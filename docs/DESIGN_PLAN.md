# RuneShell / 符令迷城 设计与开发计划

## 文档定位
- 这是项目的长期设计与实施主文档。
- 目标是统一产品方向、系统设计、数据结构演进、内容生产方式和迭代计划。
- 后续所有推荐决策默认以这份文档为准；如有变更，直接更新本文件，不额外分散到多份临时记录。

## 当前状态
- 已完成可运行 v1 骨架。
- 当前已有：2 个职业、主线 3 幕、无尽模式、地图节点流转、战斗基础规则、遗物/药水/装备、JSON 内容驱动、TUI 主流程、profile/run 存档、smoke 测试、交叉构建。
- 已完成第一轮战斗深化：相对目标结算、精英/Boss 被动窗口、`focus/frail/burn/regen/guard` 状态、状态展示、手牌弃牌回合流、两职业扩展卡池、精英/Boss 新机制。
- 已完成第二轮装备与事件成型：装备比较/替换界面、奖励/商店/事件统一走装备决策链、装备估值与 smoke 自动换装、非起始装备奖池、19 件装备池、8 个偏构筑事件模板、`lose_hp` 局外效果。
- 已完成第三轮试玩优化：战斗双栏信息 UI、战斗内牌堆/效果查看、篝火定向升级、商店付费删牌、缺失升级数值补全，以及面向多人协作的承伤/回合投票骨架。
- 已完成第四轮终端适配优化：主页面改为按真实窗口宽高渲染，战斗页在窄宽度下自动切换为纵向堆叠与战线网格，图鉴/成长/商店分组等页面改为视口优先布局并对长文本换行。
- 已完成第五轮构筑表达深化：卡牌现已支持“附加效果/词缀”体系，且不再局限于单个 `reply` 机制。
- 当前卡牌附加效果系统已支持 3 类持续范围：`run`（本局永久）、`combat`（下场战斗）、`turn`（下场战斗的当前回合）。
- `reply` 已被收敛为一个具体可挂载的卡牌附加效果示例；系统本身已经支持把任意既有效果列表挂到卡牌上，例如“使用时额外抽 1”“使用时额外获得 1 能量”。
- 内容层已新增 `augment_card` opcode，支持事件/后续工坊/特殊商店把一组效果附加到指定卡牌，并支持 `choose`、`all`、`choose_upgradable`、`all_upgradable` 等选择器。
- 单机主流程已接通“事件 -> 装备确认 -> 选牌 deck action -> 对目标卡附加效果”的完整链路；inspect、战斗展示、图鉴/列表摘要也会显示卡牌附加效果描述。
- 商店现已接通一批“工坊 service”入口，能够复用同一套 deck action 选牌流程，把 `augment_card` 作为付费构筑服务而不是再写一套专用 UI/存储逻辑。
- smoke/自动模拟已开始理解卡牌附加效果的价值评估与事件选牌；当前项目重点已经从“仅能跑通一局”提升到“能稳定承载更多 build 定向系统”。
- 当前工作区还包含一轮更激进的联机分座位状态拆分与联机 UI 强化改造，方向是“共享战场 + 每个 seat 自己的手牌/牌堆/药水/奖励/事件/商店状态”；该分支应视为已进入联调回归阶段，是否完全稳定以最新 `go test ./...` 结果为准。
- 当前版本的重点是“可跑通一局”和“架构正确”，不是数值平衡或内容量。

## 产品目标
### 核心目标
- 做成终端可玩的单机卡牌肉鸽，体验方向接近《杀戮尖塔》，但在装备、结算链路和局外养成上形成自己的差异。
- 保持 Go 单二进制、跨平台、无复杂外部依赖、可长期扩展。

### 设计原则
- 系统先于内容：先保证规则引擎和数据模型稳定，再扩内容。
- 数据驱动优先：职业、卡牌、敌人、事件、遗物、药水、装备尽量通过 JSON 扩展。
- 可测优先：战斗、地图、奖励、存档必须可复现、可断言。
- 增量复杂度：先做“少而完整”的系统，再做“多而深”的内容。
- 兼容终端：交互逻辑不依赖鼠标；键盘操作必须完整可用。

## 推荐的产品方向
### 1. 战斗定位
- 保持类 STS 回合制卡牌战斗，不改成传统 RPG 指令战斗。
- 核心循环固定为：抽牌 -> 规划能量 -> 打牌 -> 敌人行动 -> 进入下一回合。
- 推荐继续保持“低随机、高信息量、强构筑感”的方向。

### 2. 装备系统定位
- 装备不是数值挂件，而是构筑层的一部分。
- 推荐保留 3 个固定槽位：`weapon`、`armor`、`accessory`。
- 推荐每件装备只承担一种明确职责：
  - `weapon` 影响输出路径。
  - `armor` 影响承伤/格挡/生存。
  - `accessory` 影响资源、抽牌、状态、循环。
- 推荐后续避免出现“全能装备”，否则会挤压卡牌构筑空间。

### 3. 局外养成定位
- 推荐做“轻养成、重解锁”，不要做重数值碾压。
- 局外成长优先级：
  - 解锁职业
  - 解锁卡池/遗物池/事件池
  - 小幅起始加成
  - 起始装备池选择
- 不推荐把局外养成做成大量永久属性堆叠，否则会破坏肉鸽单局决策价值。

### 4. 主线与无尽模式关系
- 主线模式负责稳定节奏、叙事框架和 Boss 设计。
- 无尽模式负责长线构筑、循环成长和数值挑战。
- 推荐规则共享 80% 以上，只让“增长规则”和“奖励节奏”不同。

## 目标架构
## 模块划分
- `cmd/cmdcards`
  - CLI 入口。
- `internal/app`
  - 页面状态机、输入分发、流程跳转。
- `internal/ui`
  - 纯展示层，负责字符串渲染、面板布局、状态可视化。
- `internal/engine`
  - 规则内核，负责战斗、地图、奖励、事件、商店、无尽缩放、局外成长结算。
- `internal/content`
  - 内容定义、JSON 加载、校验、内容索引。
- `internal/storage`
  - profile/run 存档与未来版本兼容迁移。

### 推荐架构约束
- `ui` 不直接写规则。
- `app` 不直接硬编码内容数值。
- `content` 不包含运行时行为函数，只提供定义与校验。
- `engine` 不依赖 TUI 组件，只产出状态与结果。

## 规则系统设计
### 1. Effect 体系
- 继续使用 typed opcode 模型，不在 JSON 中嵌入脚本。
- 推荐把后续 effect 分三类维护：
  - 即时效果：`damage`、`block`、`draw`、`heal`
  - 状态效果：`apply_status`、`cleanse_status`、`tick_status`
  - 规则修饰：`modify_damage`、`modify_taken_damage`、`modify_cost`、`modify_draw`
- 当前已落地的构筑修饰类扩展：
  - `reply`
  - `repeat_next_card`
  - `augment_card`
- `augment_card` 的定位不是一次性结算，而是把一组现有 `content.Effect` 挂到牌上；后续凡是“让某张牌获得额外效果”的系统，优先复用它而不是新增专用 opcode。
- 推荐新增但暂缓实现的 opcode：
  - `discard`
  - `scry`
  - `exhaust_random`
  - `duplicate_card`
  - `refund_energy`
  - `gain_temporary_block`
  - `spawn_token_card`

### 2. 状态系统
- 推荐把状态统一整理为可扩展表，而不是零散 if/else。
- 推荐第一批正式支持的状态：
  - `strength`
  - `weak`
  - `vulnerable`
  - `thorns`
  - `focus`
  - `frail`
  - `burn`
  - `shielded`
- 每个状态必须明确：
  - 叠加规则
  - 持续时间
  - 回合触发点
  - 是否可驱散

### 3. 结算顺序
- 推荐固定为：
  1. 卡牌基础值
  2. 卡牌升级修正
  3. 施法者状态修正
  4. 遗物/装备被动修正
  5. 临时战斗效果修正
  6. 目标状态修正
  7. 护甲/格挡吸收
  8. 反伤/后触发
- 后续新增效果都必须声明自己挂在哪个阶段。

### 4. 敌人 AI
- 当前为固定意图循环，推荐先保留。
- 第二阶段再扩成“带权重的意图池 + 条件意图”。
- 推荐不要太早做复杂 AI；先靠高质量敌人模板与 Boss 机制提升深度。

## 内容设计规划
### 1. 职业规划
- 当前：`铁卫`、`秘术师`
- 推荐下一阶段新增 2 个职业：
  - `游侠`
    - 核心：弃牌、连击、暴击、弱点打击
  - `工匠`
    - 核心：临时装置、消耗牌、装备联动、状态转换

### 2. 卡牌内容目标
- 每个职业的内容目标分阶段推进：
  - Phase 1：12 起始牌 + 12 职业池
  - Phase 2：20-24 职业池
  - Phase 3：30+ 职业池，形成 3 套以上稳定构筑方向
- 推荐每个职业至少形成 3 条明确流派：
  - 铁卫：格挡反击 / 力量爆发 / 重甲压制
  - 秘术师：法术连锁 / 抽牌循环 / 易伤爆发
  - 游侠：连击 / 弃牌 / 中毒
  - 工匠：构件 / 装备改造 / 能量机关

### 3. 装备内容目标
- 推荐装备分 4 个稀有度：`starter`、`common`、`rare`、`legendary`
- 推荐装备条目结构后续补充：
  - 基础效果
  - 条件触发
  - 套装标签
  - 价值评分
- 推荐长期目标：
  - 每槽位至少 15 件装备
  - 形成职业偏好与中立装备共存的池子

### 4. 遗物与药水
- 遗物负责长期战斗节奏改变。
- 药水负责临场解题，不负责长期 build。
- 推荐比例：
  - 遗物偏“被动规则改变”
  - 药水偏“即时反转局面”

### 5. 事件设计
- 推荐事件模板分类：
  - 风险换收益
  - 低风险稳定收益
  - Build 定向强化
  - 特殊商人/特殊工坊
  - Boss 前铺垫事件
- 不推荐事件只给纯金币或纯回血，应该更多影响构筑。
- 当前事件设计已经正式支持“选择一张牌并赋予额外效果”的模板；后续这应成为工坊/文书馆/附魔台/祝福类事件的标准实现手段之一。

## 地图与流程设计
### 主线模式
- 固定 3 幕。
- 每幕固定 15 层，末层 Boss。
- 推荐每幕增加主题：
  - Act 1：学会核心系统，敌人机制直白
  - Act 2：中期构筑成型，出现反制敌人
  - Act 3：高压、多阶段、惩罚失误

### 无尽模式
- 复用第 3 幕后的内容模板并滚动缩放。
- 推荐新增无尽专属规则，而不是只做数值膨胀：
  - 幕间“腐化词缀”
  - 精英双遗物奖励
  - 稀有事件池
  - Boss 词缀增强

### 节点密度建议
- 怪物：主干节点
- 事件：每幕 3-4 次
- 商店：每幕 2 次左右
- 精英：每幕 2-3 次
- 篝火：每幕 2-3 次
- Boss：每幕 1 次

## 局外成长设计
### 推荐成长树结构
- `战术树`
  - 初始金币
  - 初始生命
  - 额外药水位
- `收藏树`
  - 解锁职业
  - 解锁稀有遗物池
  - 解锁特殊事件
- `工坊树`
  - 起始装备可选池
  - 装备掉落质量
  - 装备商店刷新质量

### 推荐资源设计
- 只保留 1 个主货币，避免系统噪音。
- 主货币来源：
  - 通关奖励
  - 幕进度奖励
  - 特定事件
- 推荐后续再考虑“成就点”这类第二货币，不在近期计划内。

## UI 与交互计划
### 保持方向
- 继续使用全屏 TUI。
- 默认中文文本。
- 支持方向键、`hjkl`、数字键、回车、`q`、`?`。

### UI 下一阶段重点
- 战斗页保持“宽屏分栏、窄屏堆叠”的响应式布局：
  - 我方/敌方战线在窄宽度下改为多行网格，不允许单行横向溢出
  - 顶部情报区在宽度不足时自动纵向堆叠
  - 日志与说明文本必须按面板内容宽度显式换行
- 多人模式入口直接集成到主菜单：
  - 主菜单提供明确的“多人模式”入口，不要求玩家先记住 `host` / `join` 命令
  - 进入后拆成“创建房间 / 加入房间”两条中文流程，并在页面里直接说明下一步该做什么
  - 创建房间页默认给出本机测试地址、局域网地址参考、端口建议和重连提示，让第一次玩的人也能独立完成联机
- 联机会话保持与单机一致的 TUI 体验：
  - 成功创建或加入房间后，不再切回旧的纯控制台刷屏界面，而是继续留在 Bubble Tea 主程序内
  - 房间页统一展示阶段摘要、房间提示、可用命令、建议下一步、座位状态、聊天和系统日志
  - 房间页增加“建议操作”焦点区，常用动作可直接上下选择并回车执行；带参数的命令模板自动回填到底部输入框继续补全
  - “建议操作”优先显示中文动作说明，原始协议命令退居参考区，既照顾新手，也保留老玩家熟悉的命令入口
  - 中文动作说明继续按阶段细化：战斗阶段优先展示卡牌名、药水名、目标名；需要房主决定的动作直接带“房主操作/仅房主”权限前缀
  - 房间主视图顶部固定展示“当前身份 / 当前控制”摘要，明确告诉玩家自己是房主还是队员、当前阶段是否能直接操作或只能建议
  - 联机战斗交互向单机战斗靠拢：默认焦点落在战斗操作区，上下选手牌或药水、左右切有效目标、回车执行，`z` 在手牌/药水模式之间切换，`Tab` 在战斗操作与聊天输入框之间切换
  - 联机非战斗阶段也逐步采用单机式方向键主操作：地图、奖励、事件、商店、营火、装备选择、牌组操作、结算页都支持直接上下选择和回车确认，减少对文本命令和建议动作列表的依赖
  - 联机战斗焦点进一步细分为“操作区 / 检视区 / 聊天框”三段，`Tab` 循环切换，右侧检视区可用方向键切页查看概览、日志和投票状态
  - 底部保留单行命令输入，既复用现有联机协议，又保证 UI 风格和交互反馈与单机页面一致
- 列表/详情页保持“视口优先”原则：
  - 图鉴、局外成长、商店分组等双栏页在窄宽度下自动改为上下堆叠
  - 所有标题摘要、事件说明、装备描述、奖励说明都要在面板内换行或截断
- 地图页增加简化连线与路径预览。
- 奖励页增加对比信息：
  - 当前牌库里相似牌数量
  - 装备替换对比
  - 遗物来源说明

## 存档与版本策略
- `profile` 和 `run` 继续分离。
- 每次存档结构变更都增加 `version` 迁移。
- 推荐补充：
  - 自动备份最近 1 份 run
  - profile 增加统计字段
  - 保存 seed、最终 build、击杀日志用于复盘

## 数据与平衡策略
### 推荐做法
- 为卡牌、遗物、装备、敌人增加内部 `power_budget` 或同等概念字段，辅助平衡。
- 建立简单自动对局模拟，跑不同 seed 的胜率和平均时长。
- 数值迭代顺序：
  1. 保证单局可通
  2. 保证职业差异
  3. 保证卡牌选择有意义
  4. 再做稀有内容平衡

### 不推荐做法
- 早期频繁调敌人 HP 而不看奖励质量。
- 只看单张牌强弱，不看整个流派闭环。
- 同时改卡牌、装备、遗物、敌人数值，导致回归难定位。

## 多人联机下一阶段计划
### 目标
- 把当前“可完整跑局”的 LAN 联机，继续推进到“沟通顺畅、房间自治、协作构筑开始成型”的状态。
- 下一阶段优先做 3 件事：聊天、房主转移、联机专属协作卡牌/遗物。
- 默认不同时做更重的协议改造，先在现有房间模型上增量扩展，保证可测试和可回归。

### 一、房间聊天系统
#### 推荐方案
- 聊天先做成房间广播文本流，不做私聊、不做频道、不做富文本。
- 所有聊天消息都进入房间日志，但和系统日志分层显示：
  - 系统日志继续用于阶段切换、掉线、结算、权限失败。
  - 聊天日志单独标记为 `Chat`，避免被战斗/房间系统信息淹没。
- 命令形式固定为：
  - `say <text>`
  - `chat <text>`
- 本地快捷命令补一组：
  - `chatlog` 查看最近聊天
  - `log` 继续查看系统房间日志

#### 数据与协议
- `message` 新增聊天消息载荷，但仍沿用当前单 TCP 房间协议，不额外拆频道。
- `roomSnapshot` 新增：
  - `ChatLog []string`
  - `UnreadChat int`
- 服务端状态新增聊天 ring buffer，推荐上限 40-60 条。
- 聊天写盘进房间持久化，重连后仍能看到最近聊天上下文。

#### 交互与 UX
- 聊天默认不强弹窗，只在头部显示未读计数。
- 收到新聊天时，用轻量 banner：
  - `Chat from Guest: ...`
- 玩家输入 `chatlog` 时单独打印聊天，不重绘整个阶段页。

#### 验收标准
- 2-4 人房间内任意玩家都能发聊天。
- 掉线重连后仍能看到最近聊天记录。
- 聊天不会污染系统命令解析，也不会破坏当前阶段操作。

### 二、房主转移
#### 推荐方案
- 房主转移做成显式命令，不做自动竞选。
- 命令形式固定为：
  - `host <seat>`
- 只允许当前 host 在在线成员之间转移。
- 转移后立刻生效，新的 host 取得：
  - lobby 配置权
  - 地图选点权
  - 非战斗阶段共享决策权
  - 房间保存/abandon 权

#### 为什么推荐显式转移
- 自动选主在恢复房间和掉线场景里会制造歧义。
- 当前房间模型已经把大量权限绑定在 `hostID` 上，显式转移能最小代价接入。
- 这也更适合后续做 host 交接提示和审计日志。

#### 约束
- 不能把 host 转给离线座位。
- 如果房间只剩 host 自己，则拒绝转移。
- 转移后要给全房间发阶段外 banner：
  - `Host transferred to Seat 2 Guest.`

#### 持久化与恢复
- `hostID` 本来就在房间存档里，继续沿用。
- 恢复房间后，以最后一次成功转移后的 host 为准。

#### 验收标准
- lobby / map / combat / reward 等阶段都能稳定识别新的 host 权限。
- 原 host 立刻退回 member 视角，不再看到 host-only 可执行示例。
- 存档恢复后 host 身份不丢。

### 三、联机专属协作卡牌与遗物
#### 设计原则
- 联机专属内容不应该只是“单机卡牌数值上调”，而是明确利用多人规则。
- 先做少量高辨识度内容，直接围绕：
  - 承伤重分配
  - 共享资源
  - 结束回合协同
  - 多目标配合

#### 推荐首批协作关键词
- `guard_ally`
  - 给队友上格挡。
- `intercept`
  - 本回合优先替队友承受一部分伤害。
- `shelter`
  - 降低指定队友本回合承伤。
- `focus_fire`
  - 如果本回合已有队友攻击过该目标，则追加收益。
- `rally`
  - 全队获得小额格挡、抽牌或能量类收益。
- `finisher_link`
  - 队友本回合已打出若干攻击后增强。

#### 推荐首批联机专属卡牌
- 铁卫
  - `协防架势`：给自己与一名队友格挡，并赋予 1 层 `intercept`。
  - `掩护推进`：本回合你多承受队友的部分伤害；若成功拦截则下回合获得力量。
  - `战线号令`：全队获得少量格挡，若所有在线玩家本回合都未结束，则额外抽牌。
- 秘术师
  - `法链灌注`：给一名队友临时法强/聚能，并抽 1。
  - `协同咒爆`：对一名敌人造成伤害；若本回合已有队友命中过它，则附加脆弱或燃烧。
  - `奥术共振`：全队获得小额能量或手牌质量提升效果。

#### 推荐首批联机专属遗物
- `誓约盾徽`
  - 你给队友施加格挡时，额外 +2。
- `并肩护符`
  - 回合开始时，若有队友格挡为 0，则其获得少量格挡。
- `围猎纹章`
  - 同一回合内两个不同玩家命中同一敌人后，第二次命中追加伤害。
- `战场信标`
  - 所有在线玩家都结束回合后，下一回合全队获得少量起始格挡。

#### 系统实现建议
- 先不引入“每个玩家独立手牌和能量池”的重构前提下，就能先落地一批协作内容。
- 优先复用现有：
  - `ally` 目标类型
  - `taunt / sheltered` 预留状态
  - 共享 end-turn vote
- 建议新增的最小 opcode / 状态：
  - `grant_intercept`
  - `modify_ally_taken_damage`
  - `if_target_damaged_by_team_this_turn`
  - `if_all_connected_ready`

#### 平衡约束
- 协作牌强度允许略高于普通牌，但必须绑定明确条件，不能在单人逻辑里同样超模。
- 联机专属遗物优先放在精英/Boss/特殊事件，不直接灌满普通奖励池。

### 实施顺序
1. 先做聊天系统。
2. 再做房主转移。
3. 最后做首批联机专属协作卡牌与遗物。

- 多人战斗补了“全体友军 combat_start 被动延后覆盖 ally”的逻辑，避免团队开场遗物只作用在 leader 身上。
- 聊天未读数不做纯本地假状态，而是按 seat 维持服务端 `chatSeen` 游标；`chatlog` 会同时触发已读同步。
- 房主转移改为请求-确认流：host 发起、目标 seat `accept-host` / `deny-host`，发起者可 `cancel-host`。
- 联机 UI 侧对多人专属内容的推荐方案定为“显式标签 + 阶段摘要”双层呈现，而不是只在列表里混排。
- 多人协作遗物的“同回合不同玩家配合”触发，基于 `CombatCoopState` 中的行动 seat 跟踪，不依赖客户端本地推断。
- 联机日志展示规则补充为“双通道 + 前缀统一”：聊天走 `Chat` 通道，房间系统消息走 `System` 通道，并统一带时间/来源前缀，避免聊天与阶段日志混淆。
- 联机专属收益反馈补充为“阶段日志 + 房间日志双写”：商店 service、事件收益、奖励选牌只要命中 `coop_only`，都要额外留下明确的 `CO-OP` 结果提示。
- 联机快照展示规则补充为“等待列表 + 座位状态摘要”双层表达：`Waiting on` 继续保留阻塞信息，同时额外渲染每个 seat 的实时状态，降低多人局沟通成本。
- 第四批 coop 内容继续限定在现有战斗 effect 与触发窗口内扩展，避免为了少量联机内容把战斗引擎改成脚本化或分叉实现。
- 第四批 coop 内容定版为“目标友军保护 + team_combo 奖励”两条线：卡牌继续加强 ally / all_allies 的承伤与压制协同，遗物继续围绕第二名不同玩家行动后的抽牌与回复收益展开。
- TUI 布局规则补充为“真实视口优先”：`app` 与 `ui` 不再强推固定最小宽度，战斗/图鉴/成长/商店分组页面都必须优先适配当前窗口，并为长文本提供换行或截断保障。
- 联机入口策略补充为“菜单直达 + 中文表单引导”：主程序内新增“多人模式”菜单项，并提供创建/加入房间的中文配置页；提交表单后再切换到联机会话，而不是要求用户离开游戏手动拼命令。
- 联机会话承载方式补充为“协议复用、界面内嵌”：继续复用现有房间快照和命令协议，但由 `app` 直接持有联机会话并在统一主题下渲染房间内容，避免单机/联机两套完全割裂的操作体验。
### 2026-03-27
- 卡牌系统新增“附加效果 / augment”层，`DeckCard` 与 `RuntimeCard` 都可持有一组额外挂载效果。
- 卡牌附加效果持续范围定版为 `run` / `combat` / `turn` 三档，分别对应本局永久、下场战斗、下场战斗当前回合。
- `reply` 被收敛为一个普通卡牌附加效果示例，而不是单独的硬编码特例；后续扩展优先复用同一套机制。
- 内容层新增 `augment_card` opcode，用于把一组已有 `content.Effect` 挂到卡牌上，并支持选牌/全体/仅可升级卡等目标选择器。
- 事件流正式支持“先做事件决策，再进入 deck action 选牌，再把额外效果赋予目标卡”的构筑强化模板。
- smoke/自动模拟已开始理解卡牌附加效果价值；后续凡是新 build 系统都要同步补 smoke 评分，避免内容能玩但自动回归不会选。
- 设计约束补充为：凡是“给牌加额外效果”的需求，默认先复用 `augment_card`，不要为单个效果再开新存储字段或临时分支逻辑。
- `augment_card` 已从事件扩到商店工坊 service；后续如果做“特殊商店 / 工坊节点”，优先复用同一套 service + deck action + persist 流程，而不是单独再造交互分支。
